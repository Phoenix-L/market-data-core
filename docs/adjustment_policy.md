# Adjustment Policy

## Purpose
Define how corporate-action adjustment is represented and delivered in `market-data-core`.

Phase 1 goal: keep policy simple and deterministic.

---

## 1) Supported adjustment modes

Initial supported modes:
1. `raw` (default and canonical)
2. `qfq` (forward-adjusted / 前复权) - optional derived view
3. `hfq` (backward-adjusted / 后复权) - optional derived view

If provider does not support `qfq`/`hfq` consistently, APIs must return explicit "unsupported" errors rather than silently approximating.

---

## 2) Canonical storage decision

Canonical storage layer stores **raw bars only**.

Reasons:
- Single source of truth.
- Avoids duplicating large datasets for each adjustment flavor in Phase 1.
- Keeps provider extraction stable while contracts are maturing.

---

## 3) How adjusted datasets are produced

Two paths are allowed:
1. **Read-time transform** (preferred default in Phase 1):
   - Load raw bars.
   - Apply adjustment factors in transform layer.
   - Return adjusted DataFrame without persisting by default.
2. **Materialized derived dataset** (optional optimization):
   - Generated by explicit build job.
   - Stored under curated layer with adjustment tag.

---

## 4) Naming conventions

Dataset identity should include adjustment mode:
- `cn_equity_1d_raw`
- `cn_equity_1d_qfq`
- `cn_equity_1d_hfq`
- `cn_equity_30m_raw`

API parameter convention:
- `adjustment: Literal["raw", "qfq", "hfq"] = "raw"`

Manifest metadata field:
- `adjustment_mode` with same literal values.

---

## 5) Read-time vs materialized guidance

Phase 1 recommendation:
- Use read-time adjustment for most workflows.
- Materialize only when a workload repeatedly requests same adjusted dataset and performance profiling justifies it.

Guardrails:
- Adjusted outputs must preserve canonical schema and key/order invariants.
- Adjustment logic must be versioned (`adjustment_algo_version`) in metadata when materialized.

---

## 6) Non-goals in Phase 1

- Tick-level adjustment handling.
- Intraday event-time reconstruction around splits/dividends.
- Provider-specific custom adjustment semantics beyond `raw/qfq/hfq`.

---

## 7) Deferred/open decisions

1. Primary factor source selection and precedence when multiple providers disagree.
2. Whether 30m adjusted bars are produced by:
   - adjusting 30m raw bars directly, or
   - deriving from adjusted daily anchors + intraday factors.
3. Whether adjusted datasets get dedicated storage SLA in shared data lake.
